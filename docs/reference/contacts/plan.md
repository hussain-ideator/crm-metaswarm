## Summary

Build the Contacts module вЂ” the second CRM entity and a direct dependent of the Companies module. Provides full CRUD via Django REST Framework on the backend and a Next.js 16 / TanStack Query frontend with list, detail, create, and edit pages. The backend reuses the same `TimestampedModel` + `SoftDeleteMixin` base classes, JWT Bearer auth, `django-filters`, and the existing `PageNumberPagination` subclass established by the Companies module. Free-text search fans out over `first_name`, `last_name`, `email`, and `phone` via a Django `Q` OR query. The `company_fk` and `owner_fk` fields are nullable FKs; because soft-delete does not fire Django's `on_delete` handler, nullifying `company_fk` when a company is soft-deleted requires an explicit Django signal (see [Design Decisions](#design-decisions)). The contacts app has no import-time or runtime dependency on leads, deals, or activities.

## Design Decisions

### Pagination

Reuse the `PageNumberPagination` subclass already established by the Companies module (`page` / `page_size` 1-based URL contract, max `page_size` = 100, response includes `count`, `next`, `previous`). No new pagination class is needed вЂ” import and reference the shared one or co-locate an identical subclass in `contacts/pagination.py` if the apps must stay fully isolated.

### Free-text Search (`?q=`)

Implemented as a `SearchFilter` or custom `filterset_class` expression using a Django `Q` OR query across four fields:

```python
Q(first_name__icontains=q) |
Q(last_name__icontains=q)  |
Q(email__icontains=q)      |
Q(phone__icontains=q)
```

Applied only when `q` is non-empty. Special characters are treated as literal substrings by `icontains` вЂ” no regex exposure, no SQL injection surface (ORM parameterises the value).

### Filtering (`?company=`, `?owner=`)

Handled by a `django-filters` `FilterSet` on `ContactViewSet`. Both filters accept a PK value. The `company` filter matches on `company_fk_id`; the `owner` filter matches on `owner_fk_id`. Both are optional and composable with the `?q=` search in the same request.

### Default Ordering

`ordering = ["last_name"]` on the model `Meta` and `ordering_fields = "__all__"` on `OrderingFilter`. Default is `last_name` ascending (FR-010). Client overrides via `?ordering=<field>` or `?ordering=-<field>`.

### Nullable FK Serialisation

`company_fk` and `owner_fk` are both `null=True, blank=True`. The serializer exposes them as two fields each:
- A writable `company_id` / `owner_id` integer field (accepts null to clear the association)
- A read-only `company` / `owner` nested object (name + id, for display without a second request)

This keeps the create/update payload flat while giving the frontend enough data to render the company link and owner label without additional round-trips.

### Soft-delete в†’ SET_NULL Edge Case for `company_fk`

**The problem**: Django's `on_delete=SET_NULL` only fires when a referenced row is *hard-deleted* from the database. The Companies module implements soft-delete by setting `is_deleted = True` on the company row вЂ” the underlying DB row remains. Django's FK machinery never detects this as a deletion, so `on_delete=SET_NULL` never triggers, and contacts retain a `company_fk` pointing to a logically-deleted company. This violates FR-017 and the spec's explicit edge-case behaviour ("company soft-deleted в†’ `company_fk` set to null").

**Resolution: Django `post_save` signal in `contacts/signals.py`**

Register a `post_save` receiver on the `Company` model that fires when `is_deleted` flips to `True`:

```python
from django.db.models.signals import post_save
from django.dispatch import receiver
from apps.companies.models import Company
from apps.contacts.models import Contact

@receiver(post_save, sender=Company)
def nullify_company_fk_on_soft_delete(sender, instance, **kwargs):
    if instance.is_deleted:
        Contact.objects.filter(company_fk=instance).update(company_fk=None)
```

Connect the signal in `ContactsConfig.ready()` inside `apps.py`. A dedicated test in `test_signals.py` verifies the behaviour end-to-end.

**Why not handle it at read time?** Filtering out soft-deleted companies at query time (e.g., serializer `to_representation`) would hide the FK without clearing it, leaving stale data in the database and making future queries inconsistent. The signal approach keeps the data model clean.

**Deferred alternative** (only if signal approach is blocked): Document the stale-FK state explicitly and suppress the company link display in the serializer by checking `company_fk.is_deleted` at read time. Mark as a known limitation to be fixed before the Deals module is started, since Deals will depend on `company_fk` integrity.

### URL State Persistence

All active query parameters вЂ” `q`, `company`, `owner`, `ordering`, `page`, `page_size` вЂ” are mirrored in the browser URL via Next.js `useSearchParams` / `router.replace`. A shared `useContactsSearchParams` hook reads from and writes to the URL so that any change (search, filter, sort, page) is immediately bookmarkable and shareable (FR-011, SC-004).

### Company Picker (Create / Edit Form)

The company field on the contact form is a searchable picker that queries only non-deleted (active) companies (`is_deleted=false`). The picker calls the existing Companies list endpoint with `?q=<typed text>` for typeahead. No contacts-owned company data is fetched вЂ” the dependency flows strictly one way (contacts в†’ companies).
