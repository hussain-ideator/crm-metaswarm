# Implementation Plan: Leads Module

**Branch**: `feat/leads-module` | **Date**: 2026-06-14 | **Spec**: [spec.md](spec.md)

**Input**: Feature specification from `agent-os/specs/leads/spec.md`

---

## Summary

Build the Leads module — the third CRM entity and the top of the dependency chain (core ← accounts ← companies ← contacts ← leads). Provides full CRUD via Django REST Framework on the backend and a Next.js 16 / TanStack Query frontend with list, detail, create, and edit pages, plus a convert-to-deal action.

The backend reuses `TimestampedModel` + `SoftDeleteMixin` base classes, JWT Bearer auth, `django-filters`, and the shared `PageNumberPagination` subclass. The `LeadSource` lookup table is seeded via a **data migration** (not a fixture) so it runs automatically on `migrate`. Two enum types are defined as Django `TextChoices` classes: `LeadStatus` (ADR-006) and `Salutation` (ADR-005). The `converted_deal_fk` field references `'deals.Deal'` as a **string** to avoid any import-time dependency on a module that does not exist yet. Status → `converted` can only be set by the dedicated convert endpoint; the serializer and viewset enforce this at two independent layers. Converted leads are read-only and undeletable. The leads app has no import-time or runtime dependency on deals or activities.

## Design Decisions

### Salutation Enum (ADR-005)

Define as a Django `TextChoices` class in `models.py`. The exact set is fixed by the spec assumption — five labelled values plus a blank option stored as the empty string:

```python
class Salutation(models.TextChoices):
    MR   = "Mr.",  "Mr."
    MS   = "Ms.",  "Ms."
    MRS  = "Mrs.", "Mrs."
    DR   = "Dr.",  "Dr."
    MX   = "Mx.",  "Mx."
    NONE = "",     "None"
```

`salutation` on `Lead` is `CharField(max_length=5, choices=Salutation.choices, blank=True, default="")`. The `NONE` member stores an empty string — no special sentinel value needed, blank is valid.

### LeadStatus Enum (ADR-006)

```python
class LeadStatus(models.TextChoices):
    NEW         = "new",         "New"
    CONTACTED   = "contacted",   "Contacted"
    QUALIFIED   = "qualified",   "Qualified"
    UNQUALIFIED = "unqualified", "Unqualified"
    CONVERTED   = "converted",   "Converted"
```

`status` defaults to `LeadStatus.NEW` at the model level (`default=LeadStatus.NEW`). No value outside this enum is accepted — DRF's `ChoiceField` validates on every write.

### `converted_deal_fk` — String FK to Avoid Import Dependency (FR-031)

The Deals app does not yet exist. Declaring the FK with a string reference defers the import until Django's app registry resolves it at startup:

```python
converted_deal_fk = models.ForeignKey(
    'deals.Deal',
    null=True, blank=True,
    on_delete=models.SET_NULL,
    related_name='converted_from_leads',
)
```

This field is inert until the `deals` app is added to `INSTALLED_APPS`. The migration that adds this field will fail unless `deals` is installed; the leads migration must declare a dependency on the deals initial migration before it is run. For Phase 1, if the deals app does not exist, this FK can be held back in a separate migration (`0003_add_converted_deal_fk.py`) that is only applied once the deals app lands. The leads app otherwise has no import of anything from `deals`.

### LeadSource Seeding — Data Migration (FR-008)

Seeding is done in `0002_seed_lead_sources.py` so it runs automatically on `python manage.py migrate` without any manual fixture-load step:

```python
def seed_lead_sources(apps, schema_editor):
    LeadSource = apps.get_model('leads', 'LeadSource')
    for name in ['Web', 'Referral', 'Cold Call', 'Email', 'Social Media']:
        LeadSource.objects.get_or_create(name=name)

def unseed_lead_sources(apps, schema_editor):
    LeadSource = apps.get_model('leads', 'LeadSource')
    LeadSource.objects.filter(
        name__in=['Web', 'Referral', 'Cold Call', 'Email', 'Social Media']
    ).delete()

class Migration(migrations.Migration):
    dependencies = [('leads', '0001_initial')]
    operations = [migrations.RunPython(seed_lead_sources, unseed_lead_sources)]
```

`get_or_create` makes the migration idempotent — safe to re-run in test teardown/restore scenarios.

### FR-012 Enforcement — Blocking `status=converted` via Standard Update

Enforcement lives in **two independent layers** so neither can be bypassed alone:

**Layer 1 — Serializer validation** (`serializers.py`):

```python
def validate_status(self, value):
    if value == LeadStatus.CONVERTED:
        raise serializers.ValidationError(
            "Status 'converted' can only be set via the convert endpoint."
        )
    return value
```

This runs on every `create` and `update` call through the standard DRF flow, including PATCH. It prevents `converted` from being written regardless of which viewset action triggered the serializer.

**Layer 2 — Read-only guard on converted leads** (`views.py`):

```python
def update(self, request, *args, **kwargs):
    lead = self.get_object()
    if lead.status == LeadStatus.CONVERTED:
        raise PermissionDenied("Converted leads are read-only.")
    return super().update(request, *args, **kwargs)

def partial_update(self, request, *args, **kwargs):
    lead = self.get_object()
    if lead.status == LeadStatus.CONVERTED:
        raise PermissionDenied("Converted leads are read-only.")
    return super().partial_update(request, *args, **kwargs)
```

Layer 1 catches the *value* `converted`; Layer 2 catches writes to a lead that *is already* converted. Together they satisfy FR-007 and FR-012 with no overlap or gap.

### FR-025 Enforcement — Blocking Delete on Converted Leads

Override `destroy()` on `LeadViewSet`:

```python
def destroy(self, request, *args, **kwargs):
    lead = self.get_object()
    if lead.status == LeadStatus.CONVERTED:
        raise PermissionDenied("Converted leads cannot be deleted.")
    lead.is_deleted = True
    lead.save(update_fields=['is_deleted', 'updated_at'])
    return Response(status=status.HTTP_204_NO_CONTENT)
```

The `destroy()` override also performs the soft-delete instead of calling `super()`, which would attempt a hard delete. This satisfies both FR-025 and FR-026 in a single, explicit method with no ambiguity.

### Convert-to-Deal Service (`services.py`)

All conversion logic lives in `convert_lead_to_deal(lead_id, requesting_user)` inside a single `transaction.atomic()` block. The service is called by the viewset action, never imported by the deals or contacts apps.

**Algorithm**:

```python
from django.db import transaction
from django.utils import timezone

def convert_lead_to_deal(lead, requesting_user):
    with transaction.atomic():
        # Guard: already converted
        if lead.status == LeadStatus.CONVERTED:
            raise ValueError("Lead is already converted.")

        company = None

        # Branch A — company_name present (FR-021a)
        if lead.company_name and lead.company_name.strip():
            from apps.companies.models import Company
            company = Company.objects.create(
                name=lead.company_name.strip(),
                created_by=requesting_user,
            )

        # Branch B — company_name blank: company remains None (FR-021a)

        # Always create Contact (FR-021)
        from apps.contacts.models import Contact
        contact = Contact.objects.create(
            first_name=lead.first_name,
            last_name=lead.last_name,
            email=lead.email or '',
            phone=lead.phone or '',
            mobile=lead.mobile or '',
            title=lead.title or '',
            company_fk=company,
            owner_fk=lead.owner_fk,
            created_by=requesting_user,
        )

        # Create Deal stub (FR-020) — string import avoids circular dependency
        from django.apps import apps as django_apps
        Deal = django_apps.get_model('deals', 'Deal')
        deal = Deal.objects.create(
            name=f"{lead.first_name} {lead.last_name}",
            amount=0,
            close_date=timezone.now().date(),
            owner_fk=lead.owner_fk,
            contact_fk=contact,
            company_fk=company,
            created_by=requesting_user,
        )

        # Update lead (FR-022)
        lead.status = LeadStatus.CONVERTED
        lead.converted_at = timezone.now()
        lead.converted_deal_fk = deal
        lead.updated_by = requesting_user
        lead.save(update_fields=[
            'status', 'converted_at', 'converted_deal_fk',
            'updated_at', 'updated_by',
        ])

        return deal, contact, company
```

**Key properties of this design**:
- `transaction.atomic()` wraps all three creates plus the lead update; any exception rolls back all of them (FR-023).
- Imports of `Company`, `Contact`, and `Deal` are local (inside the function body) — no module-level import from those apps, satisfying FR-031.
- `django_apps.get_model('deals', 'Deal')` is used for the Deal model because the deals app may not be importable at module load time while it's under development; `get_model` resolves at call time.
- Branch A / Branch B (`company_name` present vs. blank) is handled by a single `if` — no exceptions are raised in Branch B.

### `test_services.py` — Required Coverage for FR-021a

Both branches of the `company_name` conditional must have explicit test cases:

**Branch A — `company_name` present** (`test_convert_with_company_name`):
- Arrange: lead with `company_name="Acme Corp"`.
- Act: call `convert_lead_to_deal(lead, user)`.
- Assert:
  - Returns `(deal, contact, company)` — all three non-`None`.
  - `Company.objects.filter(name="Acme Corp").count() == 1`.
  - `Contact.objects.filter(first_name=lead.first_name).count() == 1`.
  - `contact.company_fk == company`.
  - `deal.name == f"{lead.first_name} {lead.last_name}"`, `deal.amount == 0`, `deal.close_date == date.today()`.
  - `lead.status == LeadStatus.CONVERTED`, `lead.converted_at` is set, `lead.converted_deal_fk == deal`.

**Branch B — `company_name` blank** (`test_convert_without_company_name`):
- Arrange: lead with `company_name=""` (and variant with `company_name=None`).
- Act: call `convert_lead_to_deal(lead, user)`.
- Assert:
  - Returns `(deal, contact, None)` — `company` is `None`, no exception raised.
  - `Company.objects.count() == 0` (no Company row created).
  - `Contact.objects.count() == 1`, `contact.company_fk is None`.
  - `deal` is created normally; lead is marked converted.

**Atomicity test** (`test_convert_rolls_back_on_failure`):
- Arrange: patch `Contact.objects.create` to raise `IntegrityError`.
- Act: call `convert_lead_to_deal(lead, user)` inside `pytest.raises(IntegrityError)`.
- Assert: `Company.objects.count() == 0`, `Deal.objects.count() == 0`, lead status unchanged.

**Idempotency guard** (`test_convert_already_converted_raises`):
- Arrange: lead already in `LeadStatus.CONVERTED`.
- Act: call `convert_lead_to_deal(lead, user)`.
- Assert: raises `ValueError` with message containing `"already converted"`.

### Convert Endpoint (ViewSet Action)

```python
@action(detail=True, methods=['post'], url_path='convert')
def convert(self, request, pk=None):
    lead = self.get_object()
    try:
        deal, contact, company = convert_lead_to_deal(lead, request.user)
    except ValueError as exc:
        return Response({'detail': str(exc)}, status=status.HTTP_400_BAD_REQUEST)
    serializer = self.get_serializer(lead)
    return Response(serializer.data, status=status.HTTP_200_OK)
```

Returns the updated lead representation so the frontend can immediately reflect read-only state and the deal link without a second GET.

### Free-text Search (`?q=`)

```python
Q(first_name__icontains=q) |
Q(last_name__icontains=q)  |
Q(email__icontains=q)      |
Q(company_name__icontains=q) |
Q(phone__icontains=q)
```

Applied only when `q` is non-empty. `icontains` parameterises the value via the ORM — no SQL injection or regex surface (FR-015, edge-cases).

### Filtering (`?status=`, `?source=`, `?owner=`)

Handled by a `django-filters` `FilterSet` (`leads/filters.py`). All three filters are optional and composable. `source` matches `source_fk_id`; `owner` matches `owner_fk_id`. Combined with the `?q=` search in the same queryset chain (FR-016).

### Default Ordering

`ordering = ["-created_at"]` on the model `Meta`. Client overrides via `?ordering=<field>` or `?ordering=-<field>`. `OrderingFilter` with an explicit allowlist (FR-017):

```python
ordering_fields = [
    "first_name",
    "last_name",
    "email",
    "company_name",
    "status",
    "created_at",
    "owner_fk",
]
```

Using an explicit list rather than `"__all__"` prevents clients from ordering by internal fields (e.g., `is_deleted`, `converted_deal_fk_id`) and makes the sortable surface intentional and auditable.

### Nullable FK Serialisation

`source_fk` and `owner_fk` each appear as two serializer fields:
- Writable: `source_id` / `owner_id` integer (accepts `null` to clear)
- Read-only: `source` / `owner` nested object (`{id, name}` / `{id, full_name}`) for display without a second request

`converted_deal_fk` serialises as a read-only nested `{id, name}` object on retrieve and is absent from create/update payloads.

### Pagination

Reuse the shared `PageNumberPagination` subclass (`page` / `page_size`, 1-based, max 100, response includes `count`, `next`, `previous`). Import from the shared location or co-locate an identical subclass in `leads/pagination.py` (FR-014).

### URL State Persistence (Frontend)

All active query parameters — `q`, `status`, `source`, `owner`, `ordering`, `page`, `page_size` — are mirrored in the browser URL via Next.js `useSearchParams` / `router.replace` through a shared `useLeadsSearchParams` hook (FR-018, SC-006).

### Read-only State (Frontend)

When `lead.status === 'converted'` the detail page renders all fields as plain text (no form inputs), hides the Edit and Delete buttons, shows a "Convert to Deal" button in a disabled/completed state, and displays a navigable link to the deal (`/deals/{converted_deal_fk.id}`). The Zod schema used by the create/edit form excludes `'converted'` from the `status` enum so the client-side picker never offers it as a selectable option (FR-012).

### `no_of_employees` Validation

`PositiveIntegerField(null=True, blank=True)` at the model level rejects negative values at the DB constraint level. The serializer adds an explicit `validate_no_of_employees` that raises `ValidationError` for values below zero to return a 400 before hitting the DB (edge-cases).

### Soft Delete Queryset

`LeadViewSet` sets `queryset = Lead.objects.filter(is_deleted=False)`. All viewset actions (list, retrieve, update, destroy, convert) operate on this filtered queryset; soft-deleted leads are invisible to all default operations (FR-013, FR-027).

## Complexity Tracking

No constitution violations to justify.

| Design choice | Why it adds complexity | Justification |
|---|---|---|
| Two-layer FR-012 enforcement (serializer + viewset) | Extra code path | Belt-and-suspenders prevents a race or bypass; spec explicitly disallows `converted` via standard update |
| Local imports inside `convert_lead_to_deal` | Non-idiomatic | Required by FR-031: no module-level import from `companies`, `contacts`, or `deals` |
| String FK `'deals.Deal'` + `get_model()` at call time | Deferred resolution | `deals` app does not exist; avoids `AppRegistryNotReady` or import errors at migration time |
| Data migration for LeadSource | Slightly more code than a fixture | Runs automatically on deploy without a separate `loaddata` step; idempotent |
| `destroy()` override combining soft-delete + converted-guard | Combines two concerns | Necessary — DRF's `DestroyModelMixin` would hard-delete; one override handles both requirements cleanly |
